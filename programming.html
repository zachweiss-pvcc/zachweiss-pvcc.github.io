Chapter 5: Programming Algorithms & Software Testing

Phases of Computer Program Development:

  1. Phase 1: Understand the problem
  2. Phase 2: Design an algorithm to solve the problem.
  3. Phase 3: Write the program code in the appropriate language based on the algorithm
  4. Phase 4: Enter the program and Save
  5. Phase 5: Execute the program
  6. Phase 6: Test the program, find errors and debug, then retest
  7. Phase 7: Document and evaluate the program

Programming Structures:

  • Sequential structure: Top-down execution
  • Decision structures: if/then/else
  • Iterative (Looping):
    • while loop: continue looping as long something is true
    • repeat/until loop: continue looping until something is true
    • for loop: loop given number of times
  • Recursive structures: Repeat a set of instructions as a subtask of itself
    • Repetition continues until:
      • Degenerative case, which is a termination condition, is met (degenerative case is also called a "base case")

Two Classic Programming/Database Problems:

  1. Sorting:
    • Descending order/Ascending order
    • Sorting numbers
    • Sorting strings (in accordance with ASCII values)
  2. Searching: find specific data that meet a certain criteria
    • In database:
      • Filters: temporary search
      • Queries: search criteria are saved so they don't have to be recreated
    • Two popular search techniques:
      • Sequential search: search from top to bottom or bottom to top, one item at a time
      • Binary search:
        • Only works if data is SORTED first
        • Start in the middle, then go to half of that, etc.
  3. Two Classic Program Goals:
    • Efficiency: Program takes time and/or resources
    • Correctness: Program runs in expected ways and works for all expected conditions

    4. Software Testing and Verification

    Kinds of Tests

    • Acceptance testing: Verifying whether the whole system works as intended.
    • Integration testing: Ensuring that software components or functions operate together.
    • Unit testing: Validating that each software unit performs as expected. A unit is the smallest testable component of an application.
    • Functional testing: Checking functions by emulating business scenarios, based on functional requirements.
    • Performance testing: Testing how the software performs under different workloads.
    • Regression testing: Checking whether new features break or degrade functionality.
    • Security testing: Validate that your software is not open to hacker or other malicious types of vulnerabilities.
    • Stress testing: Testing how much strain the system can take before it fails.
    • Usability testing: Validating how well a customer can use a system or web application to complete a task.

    Software Testing is Done to Detect:

    • Architectural flaws
    • Poor design decisions
    • Invalid or incorrect functionality
    • Security vulnerabilities
    • Scalability Issues

    Testing Best Practices

    • Continuous testing: Project teams test each build as it becomes available.
    • Configuration management: Organizations centrally maintain test assets and track what software builds to test.
    • Service virtualization: Testing environments may not be available, especially early in code development.
    • Defect or “bug tracking”: Monitoring defects is important to both testing and development teams for measuring and improving quality.
    • Metrics and reporting: Reporting and analytics enable team members to share status, goals, and test results.
